这个选题现在是**可以站住的**，而且比你们前面那些“安全核验 / 心理关怀 / 遥感 Agent”更适合黑客松：它有明确用户、明确场景、明确输入输出，也符合 T1「AI × 工业现场」里“设备管理、异常预警、流程优化、帮助一线人员更快定位问题”的方向。手册也强调要做“小切口、强落地、可展示”的产品，不要只把大模型当聊天工具，而要构建任务链条式能力；你们这个“多模态输入 → 工具调用 → RAG 检索 → 故障定位 → 反馈学习”的结构正好符合。

我建议你们把项目正式定义成：

# **机修老师傅 MasterFix：会从维修反馈中进化的多模态故障定位 Agent**

一句话：

> 面向工厂维修员，维修员拍设备、录声音、说故障现象后，Agent 自动调用音频、视频、手册检索、历史工单等工具，给出故障原因 Top 3、排查步骤、风险提示和工单；维修完成后，老师傅反馈真实原因，系统把这次维修沉淀成新的知识和技能，让下一次诊断更准。

最核心的产品表达不是“AI 会修设备”，而是：

> **把老师傅的隐性经验，变成工厂可复用、可检索、可进化的维修大脑。**

## 1. 这个方向最强的痛点是什么？

你们不要只讲“维修慢”。这个太普通。要讲：

# **工厂真正缺的不是手册，而是“能把现象、声音、画面和经验串起来的人”**

很多工厂并不是没有设备手册，也不是没有工单系统。问题是：

* 手册是几百页 PDF，年轻维修员不知道查哪一页；
* 历史工单写得很乱，很多经验在老师傅脑子里；
* 设备故障往往不是一句报警代码能说明白，而是“声音变了、震动变了、温度变了、画面有异常”；
* 老师傅能凭经验判断“这个声儿不对，先查轴承/皮带/润滑/联轴器”，但这种判断很难进入数据库；
* 维修完成后，真正原因、排查路径、踩过的坑，经常没有被结构化记录。

所以你们要把痛点说成：

> **维修知识不是缺资料，而是缺闭环。**
> 现场有声音、有视频、有手册、有工单、有老师傅经验，但它们没有被统一组织起来。新人每次都像第一次修，老师傅每次都要重复讲，企业的维修知识没有随着每一次维修自动变聪明。

这个比“故障定位慢”更有力。

## 2. 市场和前景：为什么有人会买？

这件事的付费逻辑比较清楚，因为它直接影响**停机时间、维修效率、人员培养、经验传承和备件采购**。

中国制造业场景足够大。第五次全国经济普查显示，2023 年末中国制造业企业法人单位为 **404.8 万个**，制造业从业人员 **10481.4 万人**。这意味着设备维修、巡检、故障排查不是小众需求，而是覆盖大量工厂的一线刚需。([统计局][1])

工厂也确实有预算入口。MRO 工业品采购服务本来就是企业持续支出的环节，毕马威白皮书显示，2024 年中国 MRO 工业品采购服务市场规模达到 **3.7 万亿元**，但数智化采购渗透率只有 **9.8%**。你们的产品虽然不是 MRO 电商，但它能把“故障定位 → 备件判断 → 工单 → 采购申请”串起来，所以可以挂在设备部、维修部、MRO 采购和数字化运维预算下面。([KPMG][2])

停机损失也足够痛。ABB 的工业可靠性调查显示，超过三分之二的工业企业至少每月经历一次非计划停机，典型企业停机成本接近 **12.5 万美元/小时**。这个数字不能直接套到中国中小工厂，但很适合在 BP 里作为“停机是一种高成本事件”的国际参考。([ABB Group][3])

还有一个更适合你们故事的点：技能人才断层。人社部相关数据显示，截至 2021 年底我国高技能人才缺口达 2000 万人；到 2025 年，中国制造业十大重点领域人才需求缺口将近 3000 万人，缺口率约 48%。这能支撑你们“老师傅经验难复制、新人培养慢”的叙事。([人民网论文频道][4])

全球看，预测性维护和智能维修也是增长方向。Grand View Research 估计，全球预测性维护市场 2025 年规模为 **142.9 亿美元**，到 2033 年预计达到 **981.6 亿美元**。你们第一阶段不要直接讲“预测性维护”，而是讲“故障定位与维修知识沉淀”，但长期可以向预测性维护、备件推荐、巡检助手扩展。([Grand View Research][5])

## 3. 这个项目是否已有竞品？有，但你们可以差异化

你们不能说“我们首创 AI 维修助手”。这个领域已经有人做。

手册里就提到 Pairio：它让技术员拍摄故障设备，结合设备手册、历史工单和语音 AI 快速搜索解决方案，把故障定位从小时级压缩到分钟级；它也特别强调工业现场的真实约束，比如拍照角度不稳定、现场噪声大、文档格式老旧、网络信号差。

MaintainX 也已经有 CoPilot，官方介绍里说它可以从 OEM 手册和历史工单中生成可执行信息，帮助团队利用设备手册、创建维护文档、回答维修问题；它的新功能还会在工单、程序和资产页面里给出任务相关建议。([MaintainX][6]) Augmentir 则是更大的 connected worker 平台，覆盖一线工人的培训、作业指导、流程数字化和 AI 洞察。([Augmentir][7])

所以你们的差异化不能是“我们也能查手册”。要改成：

> **现有维修 AI 多数是“查资料 + 给建议”，我们做的是“维修闭环学习”：每一次故障处理完成后，系统都从老师傅反馈中抽取新的故障模式、新的工具调用经验和新的排查技能。**

也就是说，你们真正的创新点不是 RAG，而是：

# **Memory + Tool + Skill 的闭环进化**

---

## 4. 产品核心概念：Memory、Tool、Skill 三层

你们现在的想法已经有雏形了，但要把它讲得更清楚。

### **Memory：工厂记忆库**

Memory 不是一个简单的文档库。它应该分成四类：

| 类型   | 存什么                 | 作用     |
| ---- | ------------------- | ------ |
| 文档记忆 | 设备手册、SOP、维修规范、安全规范  | 给出权威依据 |
| 案例记忆 | 历史工单、故障现象、真实原因、维修方法 | 找相似故障  |
| 设备记忆 | 每台设备的型号、年限、历史故障、备件  | 做个体化判断 |
| 反馈记忆 | 老师傅维修后的表单、纠错、经验补充   | 让系统进化  |

关键话术：

> **普通 RAG 查的是“资料”，我们的 Memory 记的是“这家工厂怎么修过这台设备”。**

---

### **Tool：多模态工具箱**

Tool 是 Agent 可以调用的外部能力，不能只写“音频工具、视频工具”。要具体到工具定义。

| 工具类型 | 工具示例                    | 输出                  |
| ---- | ----------------------- | ------------------- |
| 音频工具 | 去噪、频谱图、MFCC/声纹特征、异常声音分类 | “800–1200Hz 有异常冲击声” |
| 视频工具 | 抽帧、部件识别、泄漏/烟雾/皮带偏移检测    | “疑似皮带偏移 / 管路渗漏”     |
| 图像工具 | OCR 铭牌识别、仪表读数识别、锈蚀/裂纹检测 | “设备型号、报警码、表压读数”     |
| 文档工具 | 设备手册 RAG、SOP 检索、历史工单检索  | “相似案例 3 条、手册第 X 节”  |
| 数据工具 | 温度/振动/电流 CSV 异常检测       | “过去 30 分钟振动趋势异常”    |
| 业务工具 | 备件查询、工单生成、风险提示          | “建议更换轴承，生成工单”       |

关键话术：

> **Agent 不是直接猜答案，而是先判断该调用什么工具，再把工具结果和工厂知识一起推理。**

---

### **Skill：维修技能库**

Skill 是你们最该强调的创新点。

Memory 解决“知道什么”，Tool 解决“能做什么”，Skill 解决“遇到什么情况，先做什么、后做什么”。

Skill 可以长这样：

```text
技能名称：电机异响初诊 Skill

触发条件：
- 设备类型：电机 / 泵 / 风机
- 症状：异响、振动、温升、电流波动

推荐工具顺序：
1. audio_denoise
2. sound_spectrum_extract
3. image_ocr_nameplate
4. manual_rag_search
5. similar_workorder_search
6. sensor_csv_anomaly_detect

判断规则：
- 如果 1x/2x 转频增强，优先考虑不平衡或联轴器问题
- 如果高频冲击明显，优先检查轴承和润滑
- 如果伴随温升，增加润滑不足/轴承磨损权重

输出：
- 故障原因 Top 3
- 验证步骤
- 安全注意事项
- 所需备件
```

你们要强调：

> **Skill 不是预写死的流程，而是可以从一次次维修反馈中被更新。**

## 5. “自动进化”要谨慎讲：不是让 AI 自己乱学

工业场景里，“AI 自动进化”容易被质疑：如果老师傅反馈错了怎么办？如果模型学坏了怎么办？如果错误建议导致事故怎么办？

所以你们要把“自动学习”改成更可信的说法：

# **人审闭环的持续学习**

也就是：

1. 维修员提交反馈；
2. Agent 把反馈结构化成“案例卡”；
3. 系统判断这是“新增案例”“纠错案例”“新工具经验”还是“新技能规则”；
4. 老师傅或设备工程师审核；
5. 通过后才进入正式 Memory / Skill；
6. 下次诊断时可引用，并显示来源。

关键话术：

> **我们不让模型直接在线改权重，而是让它持续更新工厂的知识库、案例库和技能库。所有进入正式技能库的经验都要经过老师傅确认。**

这句话很重要，能显得你们懂工业安全和可靠性。

## 6. 完整产品流程

可以按这个流程讲：

```text
维修员输入
音频 / 视频 / 图片 / 故障描述 / 传感器数据
        ↓
多模态预处理
音频去噪、视频抽帧、铭牌 OCR、报警码识别
        ↓
Agent 规划
判断设备类型、故障现象、需要调用哪些工具
        ↓
工具调用
声纹分析、图像识别、手册 RAG、历史工单检索、数据异常检测
        ↓
证据融合
把声音、画面、手册、历史案例、传感器异常放到同一张证据板
        ↓
故障定位
输出原因 Top 3、置信度、验证步骤、风险提示
        ↓
维修执行
维修员按步骤检查、更换、复位、试车
        ↓
反馈表单
真实原因、最终处理方法、是否有效、用到哪些备件
        ↓
Memory / Skill 更新
生成新案例卡、新技能规则、工具调用经验
```

你们的 Demo 只需要跑通其中一条链，比如：

> 电机异响 → 上传音频 + 视频 + 描述 → Agent 调用声纹工具和手册 RAG → 输出“疑似轴承磨损/润滑不足/联轴器偏移” → 维修员反馈“真实原因是润滑不足” → 系统生成新的案例卡和 Skill 更新。

## 7. ReAct 框架是什么？

你说的那个很常用的应该是 **ReAct**，不是前端的 React.js。

**ReAct = Reasoning + Acting**，也就是让大模型一边规划，一边调用工具，再根据工具返回结果继续下一步。原始 ReAct 论文提出让语言模型交替生成推理轨迹和动作，从而能调用外部知识库或环境来获取信息、更新计划。([arXiv][8])

你们这个项目非常适合 ReAct，因为它天然就是：

```text
Observe：用户上传音频、视频、故障描述
Reason：判断可能是什么设备、什么故障、该查什么
Act：调用音频工具 / 视频工具 / RAG / 工单检索
Observe：读取工具结果
Reason：更新故障假设
Act：继续查手册或问用户补充信息
Final：输出诊断卡和维修步骤
```

工程上建议用 **LangGraph / LangChain Agents**。现在 LangChain 的 agent 文档里，`create_agent` 是基于 LangGraph 构建的图式 agent runtime，节点可以是 model node、tools node、middleware 等，这比单纯 while-loop 更适合你们这种有状态、有工具、有反馈学习的系统。([LangChain Docs][9])

所以你们可以这样说：

> 我们的 Agent 采用 ReAct 思路，底层用 LangGraph 实现状态机：每一轮先观察现场输入，再规划工具调用，再读取工具结果，最后生成诊断；维修完成后进入 Feedback 节点，把真实维修结果写入 Memory 和 Skill。

前端如果要做网页，那才是 **React.js / Next.js**。黑客松时间紧的话，我反而建议：**前端用 Streamlit 或 Gradio 更快；Agent 后端用 LangGraph + FastAPI；展示时重点放在工具调用日志和学习闭环。**

## 10. Demo 怎么做最稳？

黑客松时间有限，不要做全。做一个“电机/泵故障定位”小场景就够。

### Demo 场景

> 焦化厂某电机运行时出现异响和温升，新维修员不知道先查哪里。他上传 10 秒声音、一段设备视频，并说：“电机声音变尖，温度比平时高。”Agent 自动调用工具，检索手册和历史工单，输出诊断建议。维修完成后，老师傅反馈真实原因是润滑不足，系统把这次经验沉淀进 Skill。

### Demo 输入

```text
音频：motor_noise.wav
视频：motor_running.mp4
文本：电机声音变尖，温度比平时高，最近两天振动也大
设备类型：A3 区 7 号电机
```

### Demo 中间过程展示

不要展示大模型隐藏思维链，展示“工具调用日志”和“证据板”：

```text
Agent 工具调用日志：

1. audio_denoise：现场噪声较高，已去噪
2. sound_spectrum_extract：高频冲击声增强，异常分数 0.78
3. video_frame_extract：抽取 5 帧关键画面
4. nameplate_ocr：识别设备为 YX3 系列电机
```

## 11. 你们的创新性怎么讲？

可以从四个层面讲。

### 创新 1：从“维修问答”到“维修闭环学习”

普通系统是问一次答一次。你们是：

```text
诊断 → 执行 → 反馈 → 抽取经验 → 更新记忆 → 更新技能 → 下一次更准
```

话术：

> **我们不是一个静态维修问答机器人，而是一个会从每一次维修结果中学习的工厂维修大脑。**

---

### 创新 2：不只更新知识，还更新“工具调用经验”

很多 RAG 产品只会把新文档塞进知识库。你们更进一步：

> 这个案例里，哪些工具有用？哪些工具没用？下次遇到类似症状，应该先听声音、先看视频，还是先查报警码？

比如：

```text
对于“电机异响 + 温升”，声纹工具优先级上升；
对于“PLC 报警码”，OCR + 手册 RAG 优先级上升；
对于“管路泄漏”，视频抽帧 + 图像识别优先级上升。
```

这就是 Skill 学习。

---

### 创新 3：把老师傅经验结构化

老师傅常说的是：

> “这个声音不对。”
> “上次也是这个毛病。”
> “先别换轴承，看看润滑。”
> “这个报警别只看代码，要看前面温度变化。”

这些话原来进不了系统。你们的 Agent 把它转成结构化知识：

```text
触发条件 → 判断依据 → 排查顺序 → 处理方法 → 适用边界
```

话术：

> **老师傅不是被 AI 替代，而是被 AI 放大。一个老师傅修过的经验，可以被整个班组复用。**

---

### 创新 4：可信输出，不做黑箱诊断

输出不能只是“故障是轴承磨损”。必须有：

```text
证据来源
相似案例
手册依据
工具结果
置信度
下一步验证方法
何时必须找人复核
```

话术：

> **在工业现场，AI 的价值不是替人拍板，而是把证据摆清楚，让维修员更快做正确验证。**

## 18. 评委问答话术

### 问：这不就是 RAG 查手册吗？

答：

> 不是。RAG 只解决“查资料”，我们解决的是“维修闭环学习”。系统不仅查手册，还会调用音频、视频、OCR、历史工单等工具，并在维修完成后从真实反馈中抽取新案例和新技能。下一次遇到相似故障时，它会知道先调用什么工具、优先排查什么原因。

---

### 问：AI 诊断错了怎么办？

答：

> 我们不让 AI 直接替人拍板。输出是故障原因 Top 3、证据来源、验证步骤和安全提示。低置信度时系统会要求补充信息或升级给老师傅。进入 Skill 库的经验也要经过老师傅或设备工程师审核，避免错误经验污染系统。

---

### 问：工厂为什么愿意用？

答：

> 因为它解决的是设备部已经在为之付费的问题：停机、重复排查、新人培养和老师傅经验流失。我们不要求工厂先做大规模数字化改造，可以从一个车间、一类设备、几十条历史工单开始试点，只要能缩短故障定位时间，就有明确 ROI。

---

### 问：没有大量数据怎么办？

答：

> 第一阶段不依赖大规模训练，而是用手册 RAG、少量历史工单和工具调用跑起来。系统可以从每一次真实维修反馈中积累案例。也就是说，它不是等数据齐了才有用，而是在使用过程中逐步变强。

---

### 问：你们的核心壁垒是什么？

答：

> 壁垒不是某一个模型，而是工厂自己的维修记忆和技能库。手册可以复制，模型可以替换，但一家工厂多年积累的故障案例、老师傅判断逻辑、工具调用经验和设备个体历史，才是越用越强的资产。
